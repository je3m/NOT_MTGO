Scripted Magic Event Language (SMEL)

1 Introduction

1.1 Objective

The objective of the Scripted Magic Event Language (SMEL) is to provide a framework for easily representing the complex card interations in Magic: The Gathering in a way that is easy for computers to understand. SMEL seeks to create a system by which programs can determine notoriously difficult card interactions correctly without having to hard-code them into the logic of the program. In event that rules or cards change, an update to card SMEL and the scripting language framework would be all that is necessary to keep the product working, without having to worry about the dreaded spaghetti code. Additionally, it simplifies what actions are and are not possible within the game. By reducing the number of possible actions that are possible, it becomes more feasible to write a program that can simulate or predict game outcomes.

1.2 Legal Disclaimer

Please note that this documentation heavily references Magic: the Gathering, which is the intellectual property of Wizards of the Coast. For this purpose, Magic: The Gathering was the optimal game to do so, as it has a rich and diverse card set that has evolved over a period of more than two decades to create a myriad of card interactions. 2 The Language

Readability should always be the key concern when expressing SMEL code. In general, text should be in all capital letters if it refers to a binary attribute, whereas key-value attributes should be in lowercase (See section 2.1 for more information). Language keywords should also be in all capitals. Note that this is simply for readability, as the language is not case sensitive.

2.1 Representing Card Attributes

In general, whenever a card cares about specific attributes of another card, these attributes will be annotated in an Attribute Definition String (ADS). ADS take the form as follows: @(a,b,c,...), where a, b, c represent different types of attributes. Note that each attribute MUST be true in order for the ADS to match a card. If an ADS should match any card matching at least one characteristic, the or operator should be used (The or operator is |, see section 2.2 for more details. Note that although once can imagine the or operator as replacing the commas, the ADS will still treat the ord attributes as a single attribute. For example, @(a|b,c) will match ac or bc, but not just a or b). There are several different types of attributes, the most basic being static attributes. Static attributes do not have values, they are merely true or not. Examples include references to a entitys state in the game such as OPPONENT_CONTROLS, IN_GRAVEYARD, IN_HAND, ON_STACK, what type of entity it is, such as PLAYER, PERMANENT, SPELL, ACTIVATED_ABILITY, TRIGGERED_ABILITY, CARD, as well as certain evergreen keyword abilities such as VIGILANCE, FIRST_STRIKE, DOUBLE_STRIKE, HASTE, etc. The other main type of attributes are key-value attribtes. These are in the form key[op]value, where key is some attribute of the entity, such as color, type, or cost. The op is just basic comparison operators, =,!=,&=,<=,>=, < and >. Note that for some operations where the attribute is non numerical, &= is true only when both the values are exactly the same (e.g. color&=UG means the card must be both blue and green, and no other color, whereas color=UG means it can be either. In non numerical cases, the symbols <= and >= represent subset or superset respectively. The value is something that the attribute can compare to. For example, if you wanted to describe a blue or green creature, the attribute string would be @(type=creature,color&=UG). This translates to match any card that is a creature AND whose colors partially overlap with Blue-Green or in other words, match any card that is a creature AND is blue or green. A complete list of all static attributes and key-value attributes can be found in section 3.0, Recognized ADS Attributes. Note that self reference is special, and uses the ~ character, which may or may not be contained within the @(~).

2.2 Syntax

2.2.1 The OR operator, |

The or operator is |. It is used for whenever any selection of the ored items would be appropriate. For example, @(type=instant|type=sorcery) would match any sorcery or instant card or token.

2.2.2 The NOT operator, !

The not operator is ! inverts something.

2.2.3 The PSUEDO operator, %

The PSUEDO operator used before a keyword will grant a psuedo version of the attribute. The a card is given the psuedo attribute, it will behave as if it had the attribute, however anything that targets creature with that attribute will not see it. For example, a creature with %VIGILANCE would behave as if it had vigilance, however it couldnt be targeted by a spell that targets creatures with vigilance. The effect %DISCARD would cause the card to be put from the hand to graveyard, but would not trigger any discard effects such as madness or Waste Not.

2.2.4 Scope Markers, { }

Curly braces represent a scope. What they actually mean depends on the context in which they are used, but generally they are used to represent where things belong. For example, in COST {a,b,c..}, they indicate that there are several costs which must all be paid before it can be activated. When used with FLIP <WHO> @(...) {a} {b} they indicate the two different outcomes that can arise frome the coin flip.

2.2.5 State Syntax, [ ]

The state braces represent the requirement for a specific gamestate. For example, a card that can only be cast if you control an island would have a restriction of [@(YOU_CONTROL, subtype=Island)]. State braces can also care about the quantity of something using a #. For example, if you needed at least 3 mountants to cast something, the card would have the restriction [@(YOU_CONTROL, subtype=Mountain)#3]. Comparison operators are also allowed, for example you control 2 or fewer other lands would convert to [@(YOU_CONTROL, type=Land, !~)#<=2]. Note that the absense of a conditional marker with a # implies #>=. If the state syntax is preceded by an entity reference (e.g. ~, $(~, TARGET), YOU, etc.) then a questionmark, it will limit the state check to just that object. For example, ~?[OVERLOADED] will check to see if the resolving spell is overloaded.

2.2.6 Effect Arguments, < >

The brackets are used to represent arguments to an argument that the SMEL enginge should careabout. For example, COUNTER takes an option argument of where to put the countered spell when it resolves. Arguments may or may not be optional, and vary on a case by case basis. Refer to the effect documentation

2.2.7 Query Operator, $

This is a special operator. The query operator will attempt to query for some value or entity and inject it when necessary. The query operator is basically how we can hack together strange cards, or cards that have dynamic effects based on how it was cast, such as X cost spells. In best practices, it should be avoided wherever possible.

For an example of the query operator in action, consider the following card:

Abandon Hope
X1B
Sorcery
As an additional cost to cast Abandon Hope, discard X cards.
Look at target opponent's hand and choose X cards from it. That player discards those cards.

We need to figure out what we paid for X when we determine how many cards we must discard. This is where the query operator comes in. $(~, XCOST) sets the query target to itself, and asks for the cost X that was paid. Using this, we can now parse Abandon hope in SMEL.

COST {DISCARD <$(~, XCOST)>} 
EFFECT  {
    PICK <$(~, XCOST), DISCARD> @(OPPONENT)
}

This will cause us to discard X as a cost, and then pick X from an opponents hand and have them discard it. As a shorthand, we can also simplify queries the same way we do state checks, only using a $ instead of a ?. For example, $(~, XCOST) could be simplified to ~$[XCOST]. Note how it still uses [] instead of ().

However, the query operator doesnt always take ~ as a parameter. Consider the following example.

Spell Syphon 
1U
Instant
Counter target spell unless its controller pays 1 for each blue permanent you control.

EFFECT {
    UNLESS <OPPONENT, {MANA <PAY, $( @(YOU_CONTROL, color=U, PERMANENT) )>} >    {
        COUNTER <GRAVE> @(SPELL)
    }
}

Some notes on how this works. UNLESS takes two effect arguments: the first is who must satisfy the condition, and the second is a scope of the condition must be met. Here the condition is paying a certain amount of generic mana, determined by how many blue permanents you control. By specifying the attribute string, the query operator understands that it should count how many entities meet these conditions, and insert that value as the result of the query. The second scope is the effect that happens if the UNLESS condidion is not met. In this example, it counters the targeted spell. Queries can also be nested if we need to get a property of a card that we are targeting, but that wont be convered here because I havent fully worked out the details yet.

2.3 Effects

There are a limited number of effects in the game, however they can be combined in many different ways, use different zones, and target different things to create. For example Counter target spell, Counter target spell, exile it, and Counter target spell, return it to hand are all fundamentally the same, except the zone the countered spell ends up in what changes. Rather than having 3 different effects spells, we could simply just have a COUNTER effect and a way to represent where the countered spell goes. We can generalize this approach to almost all aspects of the card, significantly reducing the amount of keywords the language needs to be able to understand. For a comprehensive list of all the keywords the language understands, refer to section 3.1 Keywords and Syntax. As a general rule, after any effects relating to the activation, there can be any number of EFFECT keywords which will be executed sequentially.

2.3.1 Additional Costs

Certain spells and abilities cannot be cast/activated unless an additional cost is payed. This is where the COST keyword comes into play. It should always come after state declarations (if present) but before effect declarations Consider the following example of a cost spell with an additional cost:

As an additional cost to cast Abjure, sacrifice a blue permanent. 
Counter target spell.

This would translate into:

COST { SAC @(color=U) } 
EFFECT { 
    COUNTER <GRAVE> @(SPELL) 
}

NB: SAC does not need to specify that the card is a permanent or in play, because this is implicitly handled by the SMEL language. Note that the <GRAVE> is also not necessary, as it is implied that any spell that gets countered will go to grave. However, it is added for clarity because it may be the case that countered spells go somehwere else.

2.3.2 Kicker Costs

Sometimes a player is presented with the option to pay an additional cost when they put a spell/ability on the stack. Although kicker is a specific keyword, the language uses the kick terminology to describe all optional additional costs. KICK behaves simmilar to COST, and will always come after regular cost but before effects. The general syntax is as follows: KICK <State> {cost(s)} State describes the state change (e.g. BUYBACK, KICKED). If paid, the SMEL engine will update the state of the spell on the stack, thus the part of the spell that cares about it can check if it was kicked. For example:

Forbid
1UU
BuybackDiscard two cards. (You may discard two cards in addition to any other costs as you cast this spell. If you do, put this card into your hand as it resolves.)
Counter target spell.

Will become:

"BUYBACK" {
    KICK <BUYBACK> {DISCARD <2>}
    ~?[BUYBACK] RESOLVES <HAND>    
}
EFFECT {
    COUNTER <GRAVE> @(SPELL)
}

2.3.3 Alternate Costs

Sometimes rather than pay for a spell/ability normally, an alternate cast can be paid instead. The keyword for this is ACOST <NAME> {cost(s)}. If this is paid, the enginge should assign the named alternate cost state to the spell/ability on the stack (defaults to ALT_COST_PAID). If a state syntax marker, i.e. braces, precedes the ACOST effect, then it can only be cast if those circumstances are met. Some examples follow:

Cyclonic Rift
1U
Return target nonland permanent you don't control to its owner's hand.
Overload 6U

Snuff Out
3B
If you control a Swamp, you may pay 4 life rather than pay Snuff Out's mana cost.
Destroy target nonblack creature. It can't be regenerated.

These would respectively be translated to:

"Overload" {
    ACOST <OVERLOAD> { 6U }
    EFFECT ~?[OVERLOAD] BOUNCE .@(!type=land, !YOU_CONTROL)
}
EFFECT {
    ~?[!OVERLOAD] BOUNCE @(!type=land, !YOU_CONTROL)
}

[@(YOU_CONTROL, subtype=Swamp)] ACOST {LIFE <PAY, 4>}
EFFECT {
    DESTROY <REGEN> @(color=!B)
}

NB: The effect in the Overload scope opts not to use any scope markers. A when effect is not followed by a scope marker, it is implied to be just the next parseable command. The other effect demonstrates how braces can still be used even when implied without concern.

2.3.4 Casting Restrictions

Certain cards can only be cast when certain conditions are met. These use the keyword REQUIRE and take a state brace as an argument. E.g. Cast only if you control an island becomes REQUIRE <[@(YOU_CONTROL, subtype=Island)]>.

2.4 Naming Keywords / Scopes

As you may have noticed already, sometimes we choose to label named mechanics which are not built-in effects. We do this by using a named scope. To create a named scope, simply write the name of the scope in quotes and follow it by the scope braces. Like so:

"Overload" {
    ACOST <OVERLOAD> { 1R }
    EFFECT ~?[OVERLOAD] DAMAGE <1> .@(!type=creature, !YOU_CONTROL)
}

This associates everything within the scope as being part of the Overload effect. This allows the language to recognize that these effects are associated with the ingame mechanic. Now if a card is printed that says Overload costs {2} less the engine will be able to recognize when that effect applies even though it technically doesnt know what Overload even means until it sees it on a card. It might seem trivial here, but it essentially allows you to never have to worry about fixing the enginge whenever new mechanics are introduced. It also decreases code bloat because the engines logic doesnt need to keep track of hundreds of abilities that will probably be irrelevant most of the time.

2.5 Targeting & Choosing Modes

By default, when target(s) are specified with an ADS, it is assumed that one legal target is chosen by the player controlling the spell, other effects non-withstanding. If you want to specify multiple targets, you can do so by going X*@(...). This Syntax means select X legal targets that match ..., where X is some constant or query. If a spell or ability has you choose what if affects as it resolves use . instead of *. For cards like Clever Impersonator that have you choose as it enters the battlefield but still targets, .* will equate to Choose target. If you want to choose everything instead (e.g. for cards that use each or all), *@(...) will have the effect work on everything that matches ..., that can be legally targeted by the spell/ability. .@(...) does the same thing but ignoring the targeting restriction. Lastly, if the up to a certain amount of targets can be chosen, put a - before the . or *. Do the same but with + when you have to choose at least a certain number of targets. ] For example, Choose up to 3 target creatures or players becomes 3-.*@(type=creature|type=player).

Handing the word choose is not as simple as it may seem. Some spells like Cryptic Command have you determine modes when its cast, where Turnabout has you target a player then choose what it does on resolution. Whenever a player has to choose a mode, the MODAL keyword is used. It takes an effect argument of the number of modes you are allowed to select. An example:

Cryptic Command     1UUU
Instant
Choose two 
 Counter target spell.
 Return target permanent to its owner's hand.
 Tap all creatures your opponents control.
 Draw a card.

MODAL <2> {
    EFFECT COUNTER <GRAVE> @(SPELL),
    EFFECT BOUNCE @(PERMANENT),
    EFFECT TAP *@(OPPONENT_CONTROLS, type=Creature),
    EFFECT DRAW <1> .@(YOU)
}

Effects that choose upon resolution are in a different format, and use the keyword CHOOSE, which takes an optional arguement of a player entity who chooses. Choose has you select a target first, then presents you a scope of choices to make as it resolves. The scope can have multiple choices, but the last element in the scope the effect that happens after choices are made. These choices are referenced with the query operator. An example is below:

Turnabout 2UU
Instant
Choose artifact, creature, or land. Tap all untapped permanents of the chosen type target player controls, or untap all tapped permanents of that type that player controls.

EFFECT {
    CHOOSE @(PLAYER) {
        TYPE={ARTIFACT, CREATURE, LAND},
        CMD={TAP, UNTAP},
        ${~, CHOOSE.CMD} @(type=$(~, CHOOSE.TYPE), controller=$(~, CHOOSE.TARGET),!$(CHOOSE.CMD)ED) 
    }
}

NOTE: This is not appropriate for when you have to make one choice out of many. E.g. choose a creature should just be .@(type=creature), not some choice within a choose block. NOTE: Choices are made in order they are listed, even if they resolve at once.

2.7 Triggers

Triggers use a special syntax. They use the syntax ^( --TRIGGER CONDITION-- ) [Secondary Condition] { --What it does-- }. The secondary condition indicates some other criteria that must be met in order to put it on the stack. It is optional and is usually not necessary unless an intervening if clause is present. The what it does scope functions the same scopes in EFFECT blocks. Each effect will be resolved in order, after which players will get priority. Basic triggers are in format DOMAIN>ZONE.EVENT.SOURCE SCOPE. DOMAIN is an optional qualifier that indicates who the effect will apply to. Domain can be a type of player (e.g. YOU, PLAYER, OPPONENT) or can be a reference to a player that was chosen/target earlier using the query operator (e.g. ~$[TARGET]). The ZONE indicates the current zone the entity triggering the condition is in. Zones can be restricted to a certain type of player by using a colon (e.g. OPPONENT:GRAVE is always an opponents graveyard). To limit it to the zones under the domains control, use ? as the zone qualifier (e.g. ?:GRAVE only triggers from the graveyard of the person the effect applies to). Valid zones are: FIELD, HAND, GRAVE, EXILE, DECK, SIDEBOARD, STACK. EVENT is the game event that happened, this can also have a zone qualifier syntax (e.g. STACK.OPPONENT:CAST.HAND). These vary from zone to zone, so check the documentation. SOURCE is an optional parameter in the same syntax as zone that describes where the entity that triggered the ability must have came from first (e.g. Put into graveyard from play would be GRAVE.ENTER.FIELD) Examples of trigger conditions include:

  • ^(STACK.CAST.HAND ~) Whenever this is cast from your hand
  • ^(*.PLAY @(CARD)) Whenever a card is played from anywhere. Note how stack is not used because play includes lands.
  • ^(YOU:GRAVE.ENTER.FIELD @(type="creature")) Whenever a creature is put into your graveyard from the battlefield
  • ^(OPPONENT>?:FIELD.ENTER.EXILE @(type=land)) Whenever an opponents land is exiled from play, (they do something)

Some triggers are based on game state, and use a different syntax: DOMAIN>STEP. Step can be any step in the game, qualified with a + or - before it. + indicates the start of a step, while, - indicates the end of one. Valid steps include TURN, UNTAP, UPKEEP, DRAW, MAIN, COMBAT, COMBAT_START, COMBAT_ATTACKS, COMBAT_BLOCKS, COMBAT_DAMAGE, COMBAT_END, MAIN2, END. For example, beginning of your upkeep step is +UPKEEP and end of the turn is -TURN. Not all of these are relevant for triggers, but this syntax is also used for the UNTIL command and other time referencing events.

Any effect keyword can also be referenced. For example, whenever a player discards a card could be represented as ^(DISCARD @(CARD)). Whenever an opponent draws a card, that player loses 1 life would be written as ^(OPPONENT>?:DRAW) {LIFE <LOSE, 1>} whereas Whenever an opponent draws a card, each opponent loses 1 life would be written as ^(OPPONENT>OPPONENT:DRAW) {LIFE <LOSE, 1>} Notice how the domain does not change anything until the effect scope is reached.

2.7.1 On handling hooks for triggers

The engine should maintain a HashSet of hooks (known as the HookRegister), using the trigger condition text as a key and a entity UUID as a value. Hooks should only be in the HookRegister if and only if the event were met, they would trigger right away. Whenever a card changes zones or loses/gains abilities, the engine will check the card to see if it needs to update the HookRegister. That way, triggers will always be ready . Cards update their hooks whenever appropriate.
